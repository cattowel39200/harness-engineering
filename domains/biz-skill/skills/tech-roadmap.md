# 기술개발 계획

기술성숙도(TRL) 기반의 체계적 기술개발 로드맵을 수립하고 연차별 마일스톤, 인력 투입, 위험 대응을 계획한다.

---

## 규칙

1. **TRL 단계별 목표 명확화**: 현재 TRL과 목표 TRL을 정의하고 각 단계의 완료 기준을 설정한다
2. **연차별 정렬**: 정부사업 협약 기간(보통 1~3년)에 맞춰 마일스톤을 배치한다
3. **핵심 vs 주변 분리**: 직접 개발할 핵심기술과 외부 조달할 주변기술을 구분한다
4. **정량적 목표**: 모든 마일스톤에 측정 가능한 완료 기준(KPI)을 부여한다
5. **리스크 사전 식별**: 기술 난이도별 위험요인과 대안(Plan B)을 준비한다

---

## TRL(기술성숙도) 9단계

```
TRL 1  기초 원리 관찰/보고          ← 논문, 아이디어 단계
TRL 2  기술 개념 및 응용 분야 설정
TRL 3  핵심 기능 분석적/실험적 검증  ← PoC (개념 증명)
TRL 4  연구실 환경 기본성능 검증     ← Lab Prototype
TRL 5  유사 환경 기본성능 검증
TRL 6  유사 환경 시스템 성능 검증    ← 시제품 (Prototype)
TRL 7  실제 환경 시스템 성능 검증    ← MVP / 베타
TRL 8  실제 환경 시스템 완성         ← 상용화 직전
TRL 9  실제 환경 시스템 운용 검증    ← 양산/출시
```

**정부사업별 요구 TRL**:
- 예비창업: TRL 1~3 → TRL 5~6 (시제품 제작)
- 초기창업: TRL 3~5 → TRL 7~8 (사업화)
- TIPS: TRL 4~6 → TRL 7~9 (시장 검증)
- R&D: TRL 2~4 → TRL 6~7 (기술 검증)

## 연차별 마일스톤 수립

### 1차년도: 핵심 기술 확보

```
Q1  요구분석, 아키텍처 설계, 선행연구 조사
Q2  핵심 모듈 개발, PoC 구현
Q3  프로토타입 v1 개발, 내부 테스트
Q4  프로토타입 v2 개선, 시연 가능 수준 완성
성과물: 시제품, 기술문서, 특허 출원 1건
```

### 2차년도: 검증 및 고도화

```
Q1  베타 버전 개발, 테스트 사용자 모집
Q2  사용자 피드백 반영, 성능 최적화
Q3  보안/안정성 강화, 인증 준비
Q4  MVP 완성, 파일럿 운영 시작
성과물: MVP, 사용자 테스트 결과, 특허 등록
```

### 3차년도: 사업화 및 확장

```
Q1  상용 버전 개발, 운영 인프라 구축
Q2  정식 출시, 초기 고객 확보
Q3  기능 확장, 파트너십 체결
Q4  스케일업, 후속 버전 기획
성과물: 상용 제품, 매출 실적, 추가 특허
```

## WBS(작업분류체계) 템플릿

```
1. 기획 단계
   1.1 요구사항 분석
   1.2 기술 조사 및 벤치마킹
   1.3 시스템 아키텍처 설계
2. 개발 단계
   2.1 핵심 모듈 A 개발
   2.2 핵심 모듈 B 개발
   2.3 주변 모듈 통합
   2.4 UI/UX 개발
3. 검증 단계
   3.1 단위 테스트
   3.2 통합 테스트
   3.3 사용자 테스트 (베타)
   3.4 성능/보안 검증
4. 사업화 단계
   4.1 상용화 개발
   4.2 인프라 구축
   4.3 출시 및 운영
```

## 핵심 기술 vs 주변 기술 구분

| 구분 | 핵심 기술 | 주변 기술 |
|------|-----------|-----------|
| 정의 | 경쟁우위의 원천, 직접 개발 | 범용 기술, 외부 활용 가능 |
| 전략 | 내부 R&D, 특허 출원 | 오픈소스, API, 외주 개발 |
| 인력 | 정규직 개발자 배치 | 프리랜서, 외부 협력 |
| 예산 | 인건비 + 연구장비 | 위탁개발비 + 라이선스 |
| IP | 자사 특허/영업비밀 | 라이선스 확보 |

## 인력 투입 계획

```
역할           1차년도    2차년도    3차년도
PM             1명(상시)  1명(상시)  1명(상시)
백엔드 개발     1~2명     2명       2명
프론트엔드 개발  0~1명     1명       1명
AI/데이터       0~1명     1명       1명
디자이너        0.5명     0.5명     0.5명
QA             0명       0.5명     1명
합계           3~5명     6명       6.5명
```

**정부사업 인건비 기준**: 참여율(%) x 월급여 x 개월수. 참여율은 해당 사업에 투입하는 비율(보통 50~100%).

## 기술 위험요인 및 대응

| 위험요인 | 발생확률 | 영향도 | 대응방안 |
|----------|---------|--------|---------|
| 핵심 알고리즘 성능 미달 | 중 | 상 | 대안 알고리즘 병행 연구, 외부 자문 |
| 핵심 인력 이탈 | 중 | 상 | 기술 문서화, 2인 이상 핵심기술 공유 |
| 외부 API/플랫폼 정책 변경 | 중 | 중 | 자체 대체 모듈 개발, 멀티 플랫폼 |
| 일정 지연 | 상 | 중 | 버퍼 기간 확보, MVP 범위 조정 |
| 기술 특허 분쟁 | 하 | 상 | 선행기술 조사, 회피 설계 준비 |

## 체크리스트

- [ ] 현재 TRL 수준을 객관적으로 평가했는가
- [ ] 목표 TRL과 달성 경로가 명확한가
- [ ] 연차별 마일스톤에 측정 가능한 KPI가 있는가
- [ ] WBS가 충분히 세분화되어 일정 산출이 가능한가
- [ ] 핵심기술과 주변기술이 명확히 구분되었는가
- [ ] 인력 투입 계획이 예산과 정합성이 맞는가
- [ ] 기술 위험 Top 5와 대응방안이 준비되었는가
- [ ] 개발 일정에 테스트/버퍼 기간이 포함되었는가
