# 차별화 전략

시장분석과 비즈니스 모델을 바탕으로 "왜 이 팀이, 왜 이 방법으로" 해야 하는지를 명확히 하고, 경쟁우위를 지속시킬 진입장벽과 IP 전략을 수립한다.

---

## 규칙

1. **"왜 이 팀이?" 답변 필수**: 심사위원의 핵심 질문에 한 문장으로 답할 수 있어야 한다
2. **차별화 ≠ 기능 나열**: 단순한 기능 차이가 아닌 구조적 경쟁우위를 제시한다
3. **모방 난이도 설명**: 경쟁사가 쉽게 따라할 수 없는 이유를 논리적으로 설명한다
4. **IP 방향 포함**: 특허, 디자인권, 상표 등 지식재산 전략을 반드시 포함한다
5. **증거 기반**: 차별점마다 근거(데이터, 기술 수준, 팀 역량)를 제시한다

---

## 차별화 유형

### 5가지 차별화 유형과 적용

| 유형 | 설명 | 강점 조건 | 정부사업 어필 포인트 |
|------|------|-----------|---------------------|
| 기술 차별화 | 핵심 알고리즘, 독자 기술, 특허 기술 | 연구 역량, 논문/특허 보유 | 혁신성, 기술 사업화 |
| 비용 차별화 | 동일 기능을 낮은 원가로 제공 | 효율적 아키텍처, 자동화 | 중소기업 부담 경감 |
| 고객 경험 | UX/UI, 온보딩, 고객 성공 관리 | 도메인 전문성, 현장 경험 | 실용성, 현장 적용 |
| 데이터 | 독점 데이터, 데이터 네트워크 효과 | 고유 데이터 확보 경로 | 데이터 기반 혁신 |
| 네트워크 효과 | 사용자가 늘수록 가치 증가 | 플랫폼 모델, 양면시장 | 생태계 확장성 |

## USP (고유 판매 제안) 작성법

USP는 한 문장으로 "우리만의 핵심 차별점"을 표현한다.

### USP 공식

```
[타겟 고객]을 위한 [제품/서비스]는
[핵심 차별점]을 통해 [고객 가치]를 제공하는
유일한 [카테고리]입니다.
```

### USP 작성 기준

| 기준 | 확인 질문 | O/X |
|------|-----------|-----|
| 구체성 | 모호하지 않고 측정 가능한 차별점인가? | |
| 고객 가치 | 고객이 "그래서 나한테 뭐가 좋은데?"에 답하는가? | |
| 유일성 | 경쟁사 이름으로 바꿔도 성립하면 실패다 | |
| 간결성 | 30초 이내에 설명할 수 있는가? | |
| 증명 가능 | 데이터나 시연으로 입증할 수 있는가? | |

### USP 예시 (좋은/나쁜)

```
나쁜 USP: "AI 기반의 혁신적인 솔루션을 제공합니다"
  → 모호, 모든 회사가 쓸 수 있음

좋은 USP: "[산업] 현장의 [구체적 문제]를 [기술명]으로
          처리 시간을 기존 대비 70% 단축하는 국내 유일의 솔루션"
  → 구체적, 측정 가능, 검증 가능
```

## 진입장벽 / 해자 (Moat)

경쟁사가 쉽게 모방할 수 없는 구조적 장벽을 구축한다.

| 해자 유형 | 설명 | 구축 방법 | 소요 시간 |
|-----------|------|-----------|-----------|
| 특허/IP | 기술을 법적으로 보호 | 특허 출원, 영업비밀 관리 | 1-2년 |
| 독점 데이터 | 경쟁사가 확보 불가능한 데이터 | 고객 데이터 축적, 공공 데이터 가공 | 6개월-2년 |
| 규모의 경제 | 규모가 커질수록 원가 우위 | 고객 확보, 인프라 투자 | 2-3년 |
| 전환 비용 | 고객이 다른 서비스로 옮기기 어려움 | 데이터 락인, 워크플로우 통합 | 1-2년 |
| 브랜드/신뢰 | 시장 내 인지도와 레퍼런스 | 성공 사례, 인증, 수상 | 2-3년 |
| 네트워크 효과 | 사용자 증가가 가치 증가로 이어짐 | 플랫폼 설계, 커뮤니티 구축 | 1-3년 |

### 자사 해자 전략

```
즉시 구축 가능 (0-6개월):
  1.
  2.

중기 구축 (6개월-2년):
  1.
  2.

장기 구축 (2년 이상):
  1.
```

## IP (지식재산) 방향성

### 특허 출원 대상 검토

| 구분 | 대상 | 출원 가능성 | 우선순위 |
|------|------|------------|---------|
| 방법 특허 | 핵심 알고리즘/프로세스 | 상/중/하 | |
| BM 특허 | 비즈니스 방법 (ICT 결합 필수) | 상/중/하 | |
| 시스템 특허 | 시스템 구성/아키텍처 | 상/중/하 | |
| 디자인권 | UI/UX 디자인, 제품 외관 | 상/중/하 | |
| 상표 | 서비스명, 로고 | 상/중/하 | |

### BM 특허 가능성 판단

BM 특허는 단순한 사업 아이디어가 아닌, ICT 기술과 결합된 기술적 구성이 필요하다.

```
BM 특허 체크:
  □ 정보통신기술(ICT)을 활용하는가?
  □ 기존 방법 대비 기술적 개선이 있는가?
  □ 하드웨어 또는 소프트웨어의 구체적 구성이 있는가?
  □ 기술적 효과(속도 향상, 정확도 개선 등)를 제시할 수 있는가?
  → 4개 모두 "예"이면 BM 특허 출원 검토 가치 있음
```

### IP 로드맵

```
출원 전략:
  1순위:                    (출원 예정 시기:     )
  2순위:                    (출원 예정 시기:     )
  3순위:                    (출원 예정 시기:     )

방어 전략:
  영업비밀 관리 대상:
  저작권 등록 대상:
```

## 정부사업 심사 "혁신성" 대응 전략

정부사업 심사에서 혁신성은 보통 15-25점 배점으로, 다음을 평가한다.

### 심사 포인트별 대응

| 심사 포인트 | 대응 전략 | 증거 자료 |
|------------|-----------|-----------|
| 기술의 차별성 | 기존 기술 한계 → 자사 기술로 해결 | 성능 비교표, 특허 출원 |
| 시장 내 유사성 | 경쟁 제품 대비 기능/성능 우위 표 | 기능 비교 매트릭스 |
| 사업화 가능성 | 고객 검증, LOI, 파일럿 결과 | LoI(구매의향서), 시범운영 결과 |
| 성장 잠재력 | 확장 가능한 비즈니스 모델 | TAM/SAM/SOM, 3개년 추정 |
| 팀 역량 | 관련 분야 경력, 기술 보유 현황 | 이력서, 특허/논문, 수행실적 |

### "왜 이 팀이?" 답변 프레임

```
1. 문제 인식: "[구체적 문제]는 [현재 해결 방식]으로는 한계가 있습니다"
2. 해결 방법: "저희 팀은 [차별적 기술/방법]으로 이를 해결합니다"
3. 팀 역량: "대표의 [N]년 [분야] 경험과 [구체적 기술 역량]을 보유하고 있으며"
4. 검증 결과: "[파일럿/PoC/LoI] 등을 통해 이미 [구체적 결과]를 확인했습니다"
```

## 체크리스트

- [ ] "왜 이 팀이 해야 하는가?"에 한 문장으로 답할 수 있는가
- [ ] USP가 구체적이고 측정 가능하며 경쟁사와 대체 불가능한가
- [ ] 차별점마다 근거(데이터, 기술, 역량)가 있는가
- [ ] 경쟁사가 모방하기 어려운 구조적 이유를 설명할 수 있는가
- [ ] 해자 구축 계획이 시간대별로 수립되어 있는가
- [ ] 특허 출원 대상이 검토되었는가
- [ ] BM 특허 가능성을 판단했는가
- [ ] 정부사업 심사의 혁신성 평가 포인트에 모두 대응 준비가 되었는가
- [ ] 팀의 관련 분야 경력/역량이 차별점과 연결되어 있는가
- [ ] 고객 검증 증거(LoI, 파일럿, 설문 등)를 확보할 계획이 있는가
